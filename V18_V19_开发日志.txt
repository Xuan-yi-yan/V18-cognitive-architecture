第4天 | 2026-06-04 | P6句→词解码器 + 全链路初现
═══════════════════════════════════════════════════════════════════

  [P6 V1] SentToWordsDecoder 句→词解码
    架构: 256D → encoder(256→256→256) → 16平行头 → 16×128D
    问题: 16平行头互相不知对方输出 → 复读机

  [P6 V2] 残差串行提取 (辩论方案)
    架构: 16头串行, 每头提取后从h中减去
    h_residual = h - inv_proj(word_i)
    问题: detach导致inv_heads梯度断裂

  [P6 V3] 去detach + 0.5阻尼
    架构: h = h - 0.5 * inv_proj(word)  (梯度流通)
    问题: 梯度串联拉扯, 所有头互相干扰

  [P6 V4] 权重转置逆投影
    架构: h = h - 0.5 * W^T @ word (无额外参数)
    问题: 转置投影太猛, 后头看到负cos

  [P6 V5] inv_heads + 正交init + 正常LR
    架构: inv_proj正交初始化×0.1, detach, 0.8阻尼
    问题: 0.8阻尼太重, 头被压死

  [P6 V6] 位置嵌入 (最终方案) ⭐
    架构: 每头接收 h + pos_embed[i]
    pos_embed: randn[128,256] × 0.5
    无需残差, 无需inv_heads, 无需detach
    天然防重复: 每头起点不同
    875K参数, E50无复读, 稳定8/8
    关键突破: 最简架构解决最顽固的复读机问题

第5天 | 2026-06-05 | Gate对称锁死 + 梯度康复
═══════════════════════════════════════════════════════════════════

  问题: 所有Gate维度完全一致 (std=0.0001), 逐维分化失效
  根因: UnifiedMeta三处零/同值初始化
    1. explore_mod.weight 零初始化 → 每维信号相同
    2. p3l_act 零初始化 → sigmoid(0)=0.5 所有维同值
    3. bias init scale=0.1太小 → 输出集中在0.5

  修复:
    bias init: 0.1 → 0.5 (后改为0.1, 训练本身会推动)
    act init: zeros → randn*0.1
    explore_mod: zeros_ → xavier_uniform_(gain=0.5)
    gate_div_reward: 1/(std+eps) → log(std+eps) 防爆炸

  效果: E1 std从0.0001恢复到0.45+, 门控恢复逐维分化

  Gate目标修正 (辩论后发现):
    原始: gate最小化总loss → 随机关闭维度
    修正: gate目标=每个head独立预测准确率
    P3-L gate: 23维 → 312维 (一对一控制每个head)
    P7 gate: 由P7内部自管理, 外部不干预

  梯度链断裂修复:
    问题: p3l_loss里用 gate.item() 取Python float → 梯度断链
    修复: gate保持tensor, 梯度流回explore→meta
    效果: p3l_loss从0.56直接降到0.28 (3轮) → 之前冻结了240轮!

第7天 | 2026-06-07 | P7 sent_vec改革 + P5式±叠加
═══════════════════════════════════════════════════════════════════

  问题: sent_vec = sent_proj(word_out.mean()) 均值池化丢信息
  根因: 均值把所有词的差异抹平, 不同句子产出几乎相同的sent_vec

  修复: P5式±叠加替代mean

    可学习位置权重 sent_pos_w[nA]
    每个词位置有独立权重, 可正可负
    叠加不压缩, 保留全部词级信息

  效果: P6解码从2/8提升到6/8 (E2即6/8)

第12天 | 2026-06-12 | CUDA战争 + P1分批编码
═══════════════════════════════════════════════════════════════════

  问题: CUDA OOM反复发作 (25.76GiB allocated)
  根因:
    1. P1 word_table[6000,128]每次enc都做全量cross_attn
    2. torch.no_grad()缺失 → 梯度图残留5GB
    3. GPU驱动context残影 (进程退但CUDA不释放)

  修复:
    1. P1分批编码: 50词一批 → empty_cache()
    2. 所有编码包裹 torch.no_grad()
    3. nvidia-smi --gpu-reset 硬重置驱动
    4. 训练后CUDA context正确释放

  效果: 显存从5GB降到138MB

第17天 | 2026-06-17 | 全链路重构: 从49对数据撕开真相
═══════════════════════════════════════════════════════════════════

  凌晨发现: V19训练loss极低但预测全错。
  根因连锁:
    1. P6 cosine loss是"骗人loss" — 型/模cos=0.999但字完全不同
    2. P7偷看Bv — 训练时K/V直接传B句, 学到的是B→B恒等映射不是A→B翻译
    3. char_embed均值坍缩 — 正交向量求平均→近零向量, 所有句子sent_vec一样
    4. ABC无独立loss — 空跑不学习
    5. Gate吃4维压缩值 — 847个loss压成4个数

  千问三刀手术:
    1. P6 tied weights: 128D向量×char_weight^T→vocab logits, CE直出字
    2. P7不偷看: K/V=全词表6166字, 真正学A→词汇路由
    3. 统一训练/考试: 都用位置对齐CE, 不用全局argmax碰运气

  8小时拉锯战:
    - cos loss从1.0降到0.0001但ok=0/9 — 第一次意识到被骗
    - 换CE后49对E100就完美输出 — 确认cos loss是根因
    - 加P7后E820稳定9/9 — 确认不偷看Bv能学习
    - 逐一加回ABC→P3-L→Gate→Master Gate
    - 每个模块加回都跑49对验证, 不通过就退回

  49对→52K过渡:
    - 发现P7全词表K/V在52K上batch不能超14(注意力炸显存)
    - 加gc.collect+synchornize+empty_cache三连清理
    - 自适应batch从2→14, VRAM稳定6-7GB

  修复统计:
    本日修复致命Bug: 7个
    代码修改: train_v19_turbo.py重写约60%
    API调用: 双核辩论DeepSeek+千问3轮, 发现P5/P8/训练/考试共12处问题
    实际修复: 6处 (P5/P8旧模块问题搁置)
